# 🎯 OKR (Objectives and Key Results)

목표 설정과 실행을 연결하는 프레임워크. Andy Grove가 Intel에서 개발, John Doerr가 Google에 도입하여 세계적으로 확산.

**핵심 철학**: "아이디어는 쉽다. 실행이 전부다." — John Doerr

---

## 📐 OKR 해부학 (Anatomy)

OKR = **Objective** (목표, 무엇을) + **Key Results** (핵심 결과, 어떻게 측정할 것인가)

### 구조

```
Objective: [정성적이고 영감을 주는 목표]
├── KR1: [측정 가능한 핵심 결과] → 0.0 ~ 1.0
├── KR2: [측정 가능한 핵심 결과] → 0.0 ~ 1.0
└── KR3: [측정 가능한 핵심 결과] → 0.0 ~ 1.0
```

### 예시

```
Objective: 고객이 사랑하는 온보딩 경험을 만든다
├── KR1: 신규 사용자 7일 리텐션 40% → 70% 달성
├── KR2: 온보딩 완료율 25% → 60% 달성
└── KR3: 온보딩 관련 CS 문의 월 200건 → 50건 이하로 감소
```

```
Objective: 산업청소 시장에서 압도적 기술 리더십을 확보한다
├── KR1: Gerotto 장비 도입 후 신규 수주 5건 달성
├── KR2: 기술 제안서 수주율 30% → 55%로 향상
└── KR3: 고객 만족도 NPS 점수 60 → 80 달성
```

## ✍️ Objective 작성 규칙 (목표 작성법)

| 규칙 | 설명 | 좋은 예 | 나쁜 예 |
|------|------|---------|---------|
| **영감을 주는** (Inspiring) | 팀이 아침에 일어나고 싶게 만드는 | "업계 최고의 고객 경험을 만든다" | "고객 만족도를 올린다" |
| **정성적** (Qualitative) | 숫자는 KR에, 목표는 방향을 | "개발팀의 배포 속도를 혁신한다" | "배포 횟수를 주 3회로 늘린다" |
| **시간 제한** (Time-bound) | 보통 분기(Quarter) 단위 | "Q2에 ~를 달성한다" | 기한 없는 목표 |
| **실행 가능** (Actionable) | 팀이 직접 영향을 줄 수 있는 | "우리 팀의 코드 품질을 높인다" | "경쟁사가 시장에서 철수한다" |
| **간결** (Concise) | 한 문장, 누구나 외울 수 있는 | 1줄 | 3줄 이상의 장문 |

**Objective 수**: 조직/팀당 **3~5개** (이상적으로 3개)

## 📊 Key Result 작성 규칙 (핵심 결과 작성법)

| 규칙 | 설명 | 좋은 예 | 나쁜 예 |
|------|------|---------|---------|
| **측정 가능** (Measurable) | 숫자로 표현 | "월 매출 5억 달성" | "매출을 늘린다" |
| **구체적** (Specific) | 모호함 없이 | "신규 고객 50사 확보" | "고객을 많이 확보" |
| **시간 제한** (Time-bound) | 분기 내 달성 기한 | "6월 말까지" | "언젠가" |
| **검증 가능** (Verifiable) | Yes/No 판정 가능 | "앱스토어 평점 4.5 이상" | "앱을 개선한다" |
| **결과 중심** (Outcome) | 산출물이 아닌 결과 | "이탈률 5% 감소" | "이탈 방지 기능 출시" |

**Key Result 수**: Objective당 **2~5개** (이상적으로 3개)

### 점수 체계 (Scoring: 0.0 ~ 1.0)

| 점수 | 의미 | 해석 |
|------|------|------|
| **0.0 ~ 0.3** | 실패 (빨강) | 의미 있는 진전 없음 |
| **0.4 ~ 0.6** | 진행 중 (노랑) | 진전은 있으나 목표 미달 |
| **0.7 ~ 1.0** | 달성 (초록) | 목표 수준 도달 또는 초과 |

> **Aspirational OKR의 sweet spot은 0.6~0.7**. 매번 1.0이면 목표가 너무 쉬운 것.

## 🏗️ OKR 계층 구조 / 캐스케이드 (Hierarchy & Cascade)

```
🏢 Company OKR (회사 OKR)
    "2026년 매출 150억, 영업이익률 12% 달성"
        │
        ├── 🏬 Department OKR (부서 OKR)
        │   "영업팀: 신규 거래처 20사 확보"
        │       │
        │       ├── 👥 Team OKR (팀 OKR)
        │       │   "수도권팀: SK계열 3사 추가 수주"
        │       │       │
        │       │       └── 👤 Individual OKR (개인 OKR)
        │       │           "김대리: SK하이닉스 이천 정비 계약 체결"
        │       │
        │       └── 👥 Team OKR
        │           "울산팀: 금호석유 단가 인상 협상 완료"
        │
        └── 🏬 Department OKR
            "운영팀: 프로젝트 원가율 85% → 80%로 개선"
```

### 캐스케이드 원칙

1. **상위 OKR이 하위 OKR의 맥락(Context)을 제공** — 명령이 아닌 방향
2. **하위 OKR은 상위 KR 달성에 기여** — 정렬(Alignment)
3. **60% 하향(Top-down) + 40% 상향(Bottom-up)** — 자율성과 정렬의 균형
4. **수평 정렬도 중요** — 부서 간 의존성 반영

## 🚀 Committed vs Aspirational OKR

| 구분 | Committed OKR (약속형) | Aspirational OKR (도전형/Moonshot) |
|------|----------------------|-----------------------------------|
| **달성 기대** | 100% 달성 필수 | 60~70% 달성이 성공 |
| **자원** | 충분한 자원 할당 | 자원 부족 상태에서 시작 |
| **실패 시** | 원인 분석 + 시정 필수 | 학습의 기회 |
| **성격** | 업무 계획에 가까움 | 혁신과 도전 |
| **비율** | 전체의 ~60% | 전체의 ~40% |
| **예시** | "Q2 매출 30억 달성" | "업계 최초 AI 기반 견적 자동화 시스템 구축" |

> **Google의 원칙**: "10% 개선이 아닌 10배 개선을 목표로 하라"
> Aspirational OKR은 실패해도 기존보다 훨씬 나은 결과를 만들어낸다.

## 🔄 OKR 사이클 (Quarterly Cycle)

```
┌─────────────────────────────────────────────────────────┐
│                    분기 OKR 사이클                        │
│                                                         │
│  📝 SET (설정)          📍 CHECK-IN (점검)               │
│  분기 시작 1~2주        매주 월요일                       │
│  ┌──────────┐          ┌──────────┐                     │
│  │ 목표 수립 │ ──────▶ │ 주간 점검 │ ──┐                │
│  │ 정렬 확인 │          │ 진척 확인 │   │                │
│  │ KR 합의  │          │ 장애 제거 │   │                │
│  └──────────┘          └──────────┘   │ (12주 반복)     │
│       ▲                               │                 │
│       │                               ▼                 │
│  ┌──────────┐          ┌──────────┐                     │
│  │ 회고/반영 │ ◀────── │ 점수 산정 │                     │
│  │ 학습 정리 │          │ 결과 공유 │                     │
│  │ 다음 분기 │          │ 축하/인정 │                     │
│  └──────────┘          └──────────┘                     │
│  🔍 REFLECT (회고)      📊 SCORE (점수)                  │
│  분기 종료 후 1주       분기 마지막 주                    │
└─────────────────────────────────────────────────────────┘
```

### 각 단계 상세

| 단계 | 시기 | 활동 | 산출물 |
|------|------|------|--------|
| **Set (설정)** | 분기 시작 1~2주 | 목표 브레인스토밍, KR 정의, 상하/수평 정렬 | 확정된 OKR 문서 |
| **Check-in (점검)** | 매주 (보통 월요일) | KR 진척률 업데이트, 신호등(R/Y/G), 장애물 논의 | 주간 진척 보고 |
| **Score (점수)** | 분기 마지막 주 | KR별 0.0~1.0 점수 산정, 자기 평가 + 상호 평가 | 분기 스코어카드 |
| **Reflect (회고)** | 분기 종료 후 | 무엇이 잘 되었나/안 되었나, 다음 분기에 가져갈 교훈 | 회고 기록 |

## 📜 OKR의 역사: Andy Grove → John Doerr → Google

### Andy Grove의 Intel 시절 (OKR 기원)

- **1968년**: Andy Grove가 Intel에서 MBO(목표관리)를 변형하여 **iMBO**(Intel MBO) 개발
- **핵심 혁신**: Drucker의 MBO에서 "측정 가능한 Key Results" 개념을 추가
- **Grove의 원칙**: "나쁜 의사결정보다 의사결정 없음이 더 나쁘다"
- **Output-oriented**: 활동(Activity)이 아닌 산출(Output)에 집중

### John Doerr의 FIRE 프레임워크

John Doerr (*Measure What Matters*, 2018)가 OKR 확산의 핵심 인물. 1999년 Google에 OKR 도입.

| FIRE | 의미 | 설명 |
|------|------|------|
| **F** — Focus (집중) | 소수의 중요한 것에 집중 | 3~5개 Objective, 각 2~5개 KR |
| **I** — Inform (정보 공유) | 투명하게 공개 | 전 직원이 모든 OKR 열람 가능 |
| **R** — Reach (도전) | 높은 목표 설정 | Aspirational OKR로 한계를 넘어 |
| **E** — Execute (실행) | 추적하고 완수 | Check-in, Score, Reflect 반복 |

### Google에서의 OKR

- Larry Page & Sergei Brin이 Doerr에게 배움 (1999)
- **투명성**: 모든 직원의 OKR이 사내 인트라넷에 공개
- **평가와 분리**: OKR 점수는 인사 평가에 직접 반영하지 않음
- **70% 달성이 성공**: Aspirational OKR의 기준

## 💬 CFR: Conversations, Feedback, Recognition

OKR의 실행 엔진. 연 1회 인사평가를 대체하는 **지속적 성과관리(Continuous Performance Management)**.

### 🗣️ Conversations (대화)

| 항목 | 내용 |
|------|------|
| **빈도** | 월 1회 이상 (이상적으로 격주) |
| **형태** | 1:1 미팅, 매니저-직원 간 |
| **목적** | 목표 진척, 장애물, 성장, 커리어 논의 |
| **핵심** | 매니저가 70% 듣고, 30% 말하기 |

**대화 가이드 질문**:
- "이번 분기 OKR 중 가장 자신 있는 것은?"
- "가장 큰 장애물은 무엇인가?"
- "내가 어떻게 도와줄 수 있는가?"
- "6개월 후 어디에 있고 싶은가?"

### 📢 Feedback (피드백)

| 원칙 | 설명 |
|------|------|
| **적시성** (Timely) | 사건 발생 48시간 이내 |
| **구체성** (Specific) | "잘했어"가 아닌 "X 상황에서 Y 행동이 Z 결과를 만들었어" |
| **쌍방향** (Bidirectional) | 상→하 뿐 아니라 하→상, 동료 간 |
| **성장 지향** (Growth) | 비판이 아닌 개선 방향 제시 |

### 🏆 Recognition (인정)

| 유형 | 예시 |
|------|------|
| **동료 인정** (Peer-to-peer) | "이번 주 MVP", 감사 메시지 |
| **공개 칭찬** (Public) | 전체 회의에서 기여 소개 |
| **소규모 보상** (Small rewards) | 커피, 점심, 소액 상품권 |
| **스토리텔링** (Storytelling) | 성공 사례를 조직에 공유 |

> **인정의 원칙**: 즉시(Instant), 구체적(Specific), 공개적(Public), 가치 연결(Values-linked)

## ⚠️ 흔한 실수 (Common Mistakes)

| # | 실수 | 문제 | 해결 |
|---|------|------|------|
| 1 | **OKR이 너무 많음** | 집중 불가, 전부 50%만 달성 | 팀당 Objective 3개, KR 3개 이내 |
| 2 | **KR = 할 일 목록(Task)** | 활동을 나열, 결과 없음 | "~를 출시한다" → "~로 인해 X 지표 Y% 개선" |
| 3 | **샌드배깅 (Sandbagging)** | 쉬운 목표를 세워 1.0 달성 | Aspirational OKR 40% 이상 유지 |
| 4 | **체크인 없음** | 분기 말에 처음 확인 | 매주 15분 체크인 의무화 |
| 5 | **인사 평가에 직결** | 도전 회피, 보수적 목표 | OKR 점수 ≠ 보상. 분리 필수 |
| 6 | **Top-down 일방통행** | 자율성 없음, 동기 저하 | 40% 이상 Bottom-up으로 설정 |
| 7 | **Objective에 숫자** | 목표가 영감을 주지 못함 | 숫자는 KR에만, Objective는 정성적으로 |
| 8 | **KR이 통제 불가** | "경쟁사 매출이 감소한다" | 우리가 직접 영향 줄 수 있는 지표만 |
| 9 | **분기 중간에 포기** | 어려우면 변경/삭제 | 진짜 환경 변화 아니면 유지 |
| 10 | **개인 OKR만 존재** | 조직 정렬 없음 | 반드시 Company → Team으로 캐스케이드 |

## 📋 OKR vs KPI vs MBO 비교표

| 항목 | OKR | KPI | MBO |
|------|-----|-----|-----|
| **의미** | Objectives & Key Results (목표와 핵심 결과) | Key Performance Indicators (핵심 성과 지표) | Management by Objectives (목표관리제) |
| **개발자** | Andy Grove (Intel, 1968) | 경영학 전반 | Peter Drucker (1954) |
| **목적** | 도전적 목표 설정 + 정렬 | 현재 성과 모니터링 | 상하 합의 목표 달성 |
| **주기** | 분기 (Quarterly) | 상시 (Always-on) | 연간 (Annual) |
| **목표 수준** | 60~70% 달성이 성공 (Moonshot) | 100% 달성이 기본 | 100% 달성이 기본 |
| **투명성** | 전사 공개 | 부서/팀 단위 | 상사-부하 간 |
| **보상 연동** | 분리 권장 | 보통 연동 | 직접 연동 |
| **방향** | 양방향 (Top-down + Bottom-up) | Top-down | Top-down |
| **유연성** | 분기마다 재설정 | 연중 고정 | 연간 고정 |
| **성격** | 변화와 도전 | 안정과 관리 | 합의와 책임 |
| **적합한 조직** | 혁신/성장 추구 | 운영 안정화 | 위계적 조직 |

### 함께 사용하기

```
OKR (분기 도전 목표)
 └── "고객 경험을 혁신한다"
      ├── KR: NPS 80 달성
      │    └── KPI (상시 모니터링): 일일 NPS 추이, CS 응답시간
      └── KR: 재구매율 60%
           └── KPI (상시 모니터링): 주간 재구매율, 이탈률
```

> OKR = **어디로 갈 것인가** (방향), KPI = **지금 잘 가고 있는가** (건강), MBO = **무엇을 해야 하는가** (할당)

## 🔧 서브 커맨드 (Sub-commands)

### `/okr` — 전체 OKR 설계 (Full OKR Design)

전체 OKR 시스템을 설계한다.

**프로세스**:
1. 조직 맥락 파악 (미션, 비전, 전략 우선순위)
2. Company OKR 설정 (3~5개 Objective)
3. Department/Team OKR 캐스케이드
4. Committed vs Aspirational 분류
5. CFR 계획 수립
6. 체크인 일정 설정

**입력 필요 정보**:
- 조직명, 분기, 전략 방향
- 현재 핵심 과제/고민
- 팀 구성 (캐스케이드 시)

### `/okr:write` — OKR 작성/다듬기 (Write & Refine)

새 OKR을 작성하거나 기존 OKR을 개선한다.

**프로세스**:
1. 목표 의도 파악
2. Objective 초안 작성 → 5가지 규칙 검증
3. Key Results 도출 → 5가지 규칙 검증
4. 흔한 실수 10가지 체크리스트 통과
5. Committed/Aspirational 분류
6. 최종 OKR 출력

**품질 체크리스트**:
- [ ] Objective가 영감을 주는가?
- [ ] KR이 결과(Outcome)인가, 활동(Output)이 아닌가?
- [ ] KR에 기준선(Baseline)과 목표값(Target)이 있는가?
- [ ] 팀이 직접 영향을 줄 수 있는 KR인가?
- [ ] Objective당 KR이 2~5개인가?
- [ ] 전체 Objective가 3~5개인가?

### `/okr:review` — OKR 점수 산정 및 회고 (Score & Review)

기존 OKR의 분기 말 점수를 산정하고 회고한다.

**프로세스**:
1. 기존 OKR 확인
2. KR별 실제 달성 수치 입력
3. 0.0~1.0 점수 자동 산정
4. Objective별 평균 점수 및 상태(🟢/🟡/🔴) 판정
5. 회고 질문 가이드:
   - "목표 달성에 가장 기여한 요인은?"
   - "가장 큰 장애물은 무엇이었나?"
   - "다음 분기에 이어갈 것 / 바꿀 것은?"
   - "0.3 이하 KR의 원인은? 목표가 잘못되었나, 실행이 부족했나?"
6. 다음 분기 OKR 초안 제안

**점수 해석 가이드**:
| 상황 | 의미 | 조치 |
|------|------|------|
| 전체 평균 0.9+ | 목표가 너무 쉬움 | 다음 분기 더 도전적으로 |
| 전체 평균 0.6~0.7 | Aspirational 적정 수준 | 잘 하고 있음 |
| 전체 평균 0.4~0.6 | 진전은 있으나 부족 | 장애물 분석 필요 |
| 전체 평균 0.3 이하 | 심각한 실행 문제 | OKR 자체 또는 자원 재검토 |
